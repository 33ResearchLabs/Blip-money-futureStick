3. Auth / Waitlist / Onboarding

Auth is consolidated to two routes — /signup and /signin — both rendered by AuthHero with a User/Merchant role toggle. All legacy URLs redirect here.

PAGE: AuthHero — ROUTES /signup (signup), /signin (signin) (pages/Waitlist/AuthHero.tsx)

Legacy redirects (<Navigate replace>): /waitlist→/signup; /waitlist/user→/signup?role=user; /waitlist/merchant→/signup?role=merchant; /merchant-waitlist→/signin?role=merchant; /join-waitlist→/signup?role=user; /register→/signup?role=user; /merchant-register→/signup?role=merchant; /login→/signin?role=user; /merchant-login→/signin?role=merchant.

  • SECTION: Role toggle / cross-sell — User / Merchant segmented toggle + "Switch to…" card → set role state, sync ?role= to URL (no navigation).
  • SECTION: Embedded form — renders <Login role> (signin) or <Register role embedded> (signup).
  • ⚠️ WaitlistUserHero, UserLoginHero, MerchantLoginHero, UserLogin, UserRegister, MerchantLogin, MerchantRegister are imported in App.tsx but bound to no route (dead imports). Live forms are Login.tsx + Register.tsx.

Embedded form: Login (pages/Waitlist/Login.tsx)

  • Forms: email, password (show/hide) → authApi.login() → POST /auth/login.
    • 2FA required → 2FA modal. Missing role → external register host. Email unverified (403) → /email-verification-pending. Success → role-based dashboard: SUPERADMIN→/admin-dashboard*, merchant→/merchant-dashboard, user→/dashboard.
  • Buttons: Sign up (→ external register host), Forgot password? → /forgot-password (+?role=merchant).
  • 2FA modal forms: TOTP code → /twofa/verify-login; Recovery code → /twofa/verify-recovery. Both → login + dashboard.
  • (*Note: /admin-dashboard has no route; the admin dashboard route is /superadmin — potential dead redirect.)

Embedded form: Register (pages/Waitlist/Register.tsx)

  • Forms: email, password (strength meter), confirmPassword, referral_code (from ?ref=), reCAPTCHA → authApi.register() → POST /auth/register.
    • ROLE_ADDED → login + dashboard. Normal success → /email-verification-pending (state {email, role}). EMAIL_EXISTS → toast w/ "Log in" → external login host.
  • Buttons: Switch role (external), Sign in (external login host), Terms → /terms, Privacy → /privacy.
  • OTP modal (verifyOTP / resendVerification) — present but not triggered by current submit flow.

PAGE: ForgotPassword — ROUTE /forgot-password (pages/Waitlist/ForgotPassword.tsx)

  • Form: email → authApi.forgotPassword → POST /auth/forgot-password → in-page success. Buttons: Back to Login (external), Try different email.

PAGE: ResetPassword — ROUTE /reset-password (pages/Waitlist/ResetPassword.tsx)

  • Form: password + confirm → authApi.resetPassword(token) → POST /auth/reset-password/:token → success. Reads ?token/?oobCode. Invalid-link view → Request New Link → /forgot-password.

PAGE: VerifyEmail — ROUTE /verify-email (pages/Waitlist/VerifyEmail.tsx)

  • On mount: authApi.verifyEmail(token) → GET /auth/verify-email. Success → Go to Login (external, merchant-login if MERCHANT). Error → resend form (POST /auth/resend-verification).

PAGE: EmailVerificationPending — ROUTE /email-verification-pending (pages/Waitlist/EmailVerificationPending.tsx)

  • Buttons: Resend Email (POST /auth/resend-verification, 60s cooldown), Back to Login (external), contact support → /contact.

PAGE: RedeemTelegram — ROUTE /redeem (pages/Waitlist/RedeemTelegram.tsx) — public, auth-aware

  • Reads ?token/?otp/?role → airdropApi (getTokenInfo, getRedeemStatus, verifyLinkToken). Not-logged-in → Join Waitlist / Log In (external). OTP form → airdropApi.verifyRedeemOTP. Success → Go to Dashboard → /dashboard.

PAGE: TwoFactorAuth — ROUTE /twoFactorAuth (components/TwoFactorAuth.tsx) — authenticated

  • Enable (QR + TOTP verify → /twofa/enable, /twofa/verify-enable), recovery codes (download .txt), regenerate (/twofa/regenerate-recovery), disable (/twofa/disable). Back → navigate(-1).

MODALS: TelegramVerificationModal / TelegramVerify (components/)

  • Join channel (window.open Telegram) → enter Telegram User ID → verify (POST /telegram/verify or parent onVerify) → award points. Used by dashboards.

9. Cross-Cutting Findings & Anomalies

  • External handoff: Nearly all conversion CTAs leave the marketing SPA for app.blip.money/waitlist/*. The on-site /signup & /signin render the same forms, but the forms' own "switch/sign-in/sign-up/back" links point to external app.blip.money hosts.
  • Dead imports / routes: WaitlistUserHero, UserLoginHero, MerchantLoginHero, UserLogin, UserRegister, MerchantLogin, MerchantRegister are imported but routed nowhere. /uae links to /register-uae which doesn't exist (→ 404). Login's getDashboardForUser returns /admin-dashboard but only /superadmin is defined.
  • Inert buttons: Bounty hero buttons + "Reward Documentation" (href="#") + the Bounty submit (preventDefault only); CardPreview's BecomeMerchantPreview buttons; AppLanding store pills.
  • Mocked, no backend: ComingSoon "Notify Me" form (simulated). Most live feeds (LiveNetworkFeed, dashboards' sample activity) are client-simulated; live rates come from useP2PRate (p2prate.live) and CoinGecko.
  • Security: /admin uses a hardcoded client-side password in source. /card-creator is fully unprotected. /merchant-dashboard-preview exposes the merchant dashboard with no role guard.
  • Mixed link types: /whitepaper, /merchant, /docs are reached via both router <Link> (SPA nav) and raw <a> (full reload) in different sections.
  • Only real submitting forms: Auth (login/register/forgot/reset/verify/2FA/change-password/commit-volume), ContactUs (/auth/contact-form), quest verification modals, RedeemTelegram. Everything else is filter/demo/clipboard.
  • Data-driven content: Blog, Research, FAQ, Glossary, Compare, UseCases, Changelog, pSEO corridors all read from src/data/* files; list→detail via slug.
